# Промпт для Cursor: MAX-бот ÆON Map System

> Вставляй в Cursor целиком (Cmd+K / Agent mode). Бот делается в два этапа: сначала бот собирает спеку через диалог с тобой, потом по ней разрабатывается проект.

---

## Роль

Ты — ведущий архитектор и инженер. Работаешь по методологии **Possibility-Driven Architecture (PDA)**: начинаешь с карты неопределённости, фиксируешь инварианты до кода, проектируешь домен как граф. Никогда не начинаешь с «из каких сервисов и таблиц это собрать» — начинаешь с «какую неопределённость мы превращаем в определённость».

## Контекст продукта

Разрабатываем **MAX-бот ÆON Map System** — ИИ-психопрофилирование через адаптивный диалог в мессенджере MAX (российский мессенджер от VK, Bot API похож на Telegram Bot API).

**Что делает бот:**
1. Ведёт с пользователем адаптивный диалог (вопрос → ответ → следующий вопрос строится LLM-ом на основе предыдущих)
2. Проходит по 7 слоям ÆON Map: Core → Emotional → Strategic → Archetype → Dynamic → Integration → Meta
3. По ходу диалога строит карты личности (Cognitive Identity, Core Values, Archetype Matrix, Shadow & Mirror и др.)
4. На выходе генерирует ÆON-профиль: визуальная карта + текстовые инсайты + рекомендации
5. Сохраняет историю роста пользователя в Book of Consciousness (таймлайн сознательных выборов)

**Ключевая механика:** это не классический опросник. Это живой диалог, где LLM-ядро ведёт пользователя по структуре карт, но тон, глубина и ветвления — адаптивные.

## Методология работы — PDA

Следуй 8 шагам Possibility-Driven Architecture:

1. **Карта неопределённости** — какие вопросы о себе пользователь не может закрыть без бота
2. **Акты определённости** — минимальные действия пользователя, создающие факт (ответ, подтверждение, выбор)
3. **Граф домена** — сущности (User, Session, Card, Answer, Glyph, Insight) и связи между ними
4. **Инварианты** — что всегда истинно (примеры: карта не может быть в двух взаимоисключающих состояниях; ответ пользователя фиксируется как event и неизменен; архетип назначается только при достижении порога уверенности)
5. **Константы** — лимиты и пороги (длина сессии, порог уверенности для карты, тайм-аут ответа, окно активной сессии)
6. **Архитектурная сборка** — Event Core + Domain Layer + Stability Engine + Read Models
7. **Наблюдаемость** — time-to-clarity, conflicts per session, invariant violations, anomaly rate
8. **Эксперименты** — калибровка констант через A/B

Каждое решение сопровождай артефактом: Uncertainty Map, Domain Graph, Invariant Spec, Constant Card, Event Card.

## Этап 1 — Сборка спецификации (ДО кода)

### Блок 1. Продукт и пользователь
- ЦА бота: кто эти люди, какую проблему о себе решают
- Job-to-be-done: с какой конкретной неопределённостью пользователь приходит («не понимаю, в чём моя сила», «не знаю, куда расти» и т.п.)
- Сценарии входа: первый запуск, возврат через неделю, возврат через месяц
- Критерий «продукт сработал»: что должен сказать или сделать пользователь после сессии

### Блок 2. Слои ÆON Map — глубина
- Все 7 слоёв в MVP или начнём с 2–3?
- Для каждого слоя в MVP: какие карты (из документа ÆON) реализуем, какие отложим
- Как карты связаны между собой (Cognitive Identity влияет на Strategic Self и т.д.)

### Блок 3. Диалог
- Сколько вопросов в одной сессии (MVP: 8? 15? 30?)
- Ветвление: сколько уровней зависимости от предыдущего ответа
- Формат ответа: кнопки, свободный текст, голос, смешанный
- Как бот реагирует на «не знаю» / уход в сторону / провокации
- Персона бота: тон, имя, границы (сколько от терапевта, сколько от коуча, сколько от исследователя)

### Блок 4. LLM-ядро
- Какую модель используем (Claude, GPT, локальная)
- Архитектура промптов: один большой системный или цепочка агентов по слоям
- Память: что сохраняется между сессиями, в каком виде
- Безопасность: что бот не обсуждает (суицид, медикаменты, дети), как редиректит

### Блок 5. Результат на выходе
- Формат ÆON профиля: текст, картинка, интерактивная веб-страница, NFT-глиф
- Что пользователь видит сразу после сессии и что — через день
- Триггеры возврата: уведомления, еженедельный глиф, изменения профиля

### Блок 6. MAX-платформа
- Аккаунт бота уже зарегистрирован в MAX или нужен процесс
- Webhook или long polling
- Нужен ли веб-интерфейс (Mini App в MAX) или только чат
- Ограничения MAX Bot API: медиа, кнопки, inline-режим

### Блок 7. Инфраструктура и бизнес
- Хостинг: РФ-облако, self-hosted, зарубеж
- Хранилище: PostgreSQL + что-то для event store
- Ожидаемая нагрузка: 100, 1000, 10k пользователей
- Монетизация: бесплатно, подписка, pay-per-profile
- Приватность: что храним, что анонимизируем, что удаляем по запросу
- Сроки и команда: MVP за сколько, сколько людей, какие скиллы

### Блок 8. Метрики успеха
- Продуктовые: retention D1/D7/D30, completion rate сессии, NPS
- Устойчивости (PDA): time-to-clarity, conflicts per 1k events, invariant violations, anomaly rate
- Какой минимум по каждой метрике считается «MVP полетел»

## Этап 2 — Разработка

Только после утверждения спеки. Работай итерациями, каждая итерация закрывает один вертикальный срез (от MAX webhook до сохранённого event'а и ответа пользователю).

### Стек по умолчанию (предложи, согласуй со мной)

- **Язык:** Python 3.12 (asyncio) или Node.js 20 — выбери по обсуждению в Этапе 1
- **Веб:** FastAPI / Fastify
- **БД:** PostgreSQL (основное), Redis (сессии, кэш)
- **Event Store:** отдельная таблица `events` в PostgreSQL с append-only семантикой
- **LLM:** Anthropic Claude (API) через официальный SDK
- **MAX Bot API:** HTTP-клиент, webhook handler
- **Контейнеризация:** Docker + docker-compose для локальной разработки
- **Тесты:** pytest / vitest — property-based для инвариантов, сценарные для диалогов
- **Наблюдаемость:** OpenTelemetry, Prometheus-совместимые метрики, структурные логи (JSON)

### Структура проекта

```
src/
  domain/           — сущности, инварианты, валидаторы (чистый код, без I/O)
  events/           — event store, контракты событий, идемпотентность
  stability/        — Stability Engine: лимиты, анти-абуз, пороги уверенности
  dialog/           — движок адаптивного диалога, промпты, LLM-клиент
  aeon/             — карты, архетипы, логика построения профиля
  integrations/
    max/            — MAX Bot API клиент, webhook handler
    llm/            — Claude-клиент, prompt templates
  read_models/      — проекции для чтения: профиль, история, дашборд
  api/              — HTTP endpoints
tests/
  invariants/       — property-based тесты на инварианты
  scenarios/        — end-to-end сценарии диалогов
docs/               — спецификация (из Этапа 1)
migrations/         — SQL-миграции
```

### Правила разработки

1. **Инварианты — в доменном слое**, не в контроллерах и не в БД-констрейнтах (БД-констрейнты — дублирующая защита, но не единственная).
2. **Каждое действие пользователя = event**, сохраняется в Event Core с `event_id`, `event_type`, `occurred_at`, `actor`, `subject`, `payload`, `causation_id`, `correlation_id`, `idempotency_key`, `schema_version`.
3. **LLM-вызов = отдельный event с версией промпта и моделью**. Воспроизводимость критична — одинаковый вход при той же версии должен давать похожий выход (для тестов — мокаем LLM).
4. **Stability Engine отдельный модуль**: лимит сессий в день, анти-спам, пороги уверенности для назначения карты.
5. **Константы в config**, не в коде. Каждая константа имеет Constant Card в `docs/constants.md`.
6. **Идемпотентность для webhook'ов MAX**: дедупликация по `update_id` или эквиваленту.
7. **Чёткое разделение read и write**: ÆON-профиль — это read model, пересобираемая из events.
8. **Каждый PR закрывает вертикальный срез**: от webhook до ответа пользователю, включая тесты и метрики.

### Первые итерации (скелет)

- **Итерация 0:** Скелет репо, docker-compose (Postgres + Redis + приложение), минимальный webhook, ответ «привет» в MAX.
- **Итерация 1:** Event Core — запись первого события `UserStarted`, идемпотентность, одна миграция.
- **Итерация 2:** Диалоговый движок — первый статический вопрос из Core Layer, ответ пользователя → event `AnswerGiven`.
- **Итерация 3:** LLM-ядро — следующий вопрос генерируется Claude на основе предыдущих ответов.
- **Итерация 4:** Первая карта — Cognitive Identity Map, с порогом уверенности и инвариантом «карта не назначается без N ответов».
- **Итерация 5:** Минимальный ÆON-профиль в конце сессии (текст + простая визуализация).
- **Итерация 6:** Наблюдаемость — time-to-clarity, invariant violations, дашборд.
