## Блок 1. Продукт и пользователь

**Статус блока:** зафиксировано.

### Целевая аудитория (MVP)

Только **primary** из vibepp (secondary в MVP не целим):

- люди в точке выбора: карьера, отношения, смыслы;
- практики саморазвития, коучинга, психологии;
- предприниматели и лидеры, строящие команды.

### Job-to-be-done (главный на первый заход)

**Один фокус:** «не понимаю, в чём моя реальная сила».

Остальные формулировки из vibepp — не приоритет MVP, пока не скажешь иначе.

### Онбординг и первое касание

- Пользователь приходит **с нуля**: **нет** объяснений про ÆON, слои, карты — это внутренняя кухня продукта.
- Снаружи — **живой разговор**.
- За **2–3 сообщения** пользователь должен уловить одну мысль: **«здесь я отвечаю честно — и получаю зеркало себя»**.

### Сценарии входа

| Сценарий | Описание |
|----------|----------|
| Первый запуск | С нуля; фокус на честности и «зеркале» без терминологии ÆON (см. выше). |
| Возврат через ~неделю | _(детализируем вместе с Блоком 5/триггерами)_ |
| Возврат через ~месяц | _(детализируем вместе с Блоком 5/триггерами)_ |

### Критерий «продукт сработал»

**Не метрика:** красивые фразы пользователя (легко симулируются).

**Приоритет сигналов (лучший → слабее):**

1. **Вернулся через 7+ дней** — поведение не врёт.
2. **Поделился профилем** — социальное доверие к результату.
3. **Подтвердил инсайт** («да, это про меня») — акт определённости **внутри сессии**.

**Порог MVP (фикс в Блоке 8):** retention **D7 ≥ 25%** на запуске MVP.

## Блок 2. Слои ÆON Map — глубина

**Статус блока:** обновлено по **ADR 002** — переход на протокольный режим вычисления карт.

### MVP: какие слои и какие карты

Линейный **unlock order** прохождения (как в vibepp, `aeon_layers.mvp`):

| Слой | id | Название | Карты | Протокол / длина | Условие открытия |
|------|----|----------|--------|------------------|------------------|
| I | I | Core Layer | CognitiveIdentityMap | `methodology/Cognitive_Identity_Map_v1.md` / 12 вопросов | `session.started` |
| II | II | Emotional & Motivational Layer | AffectiveResonanceMap, CoreValuesMap | методики ещё не написаны | Layer I completed |
| IV | IV | Archetype Layer | ArchetypeMatrix | методика ещё не написана | Layer II completed |

**BehavioralPatternMap** перенесена из MVP-Core в **iter-5 / v1** (ADR 002) до появления собственной методики и протокола.

**Не в MVP (дорожная карта vibepp):**

- **v1:** слой III Strategic (StrategicSelfMap, LimitMap); слой V Dynamic AI (EnergyFlowMap, ResonanceMap).
- **v2:** слой VI Integration (CompatibilityMap, LearningMap, EnvironmentMap); слой VII Meta (AeonConsciousnessMap) — в т.ч. генерация Glyph и Book of Consciousness в терминах vibepp.

### Book of Consciousness (MVP)

По `features.core`: **каждая завершённая сессия добавляет запись** в Book of Consciousness. Отдельный сценарий «только один снимок без истории» не выбираем — таймлайн с первой же завершённой сессии.

### Сквозные связи: временно отключены (ADR 002)

Сквозной **signal routing** (один ответ → несколько `card_signal.received` для разных карт) **временно исключён** вместе с пороговой моделью. Каждая карта собирается строго по **своему протоколу**; ответы Cognitive v1 не пополняют автоматически копилки других карт.

Возврат сквозного routing будет рассмотрен после стабилизации протоколов нескольких карт; решение оформляется отдельным ADR.

### Расширение доменной модели (к Блоку 2 / domain graph)

Дополнения к каноническому графу из vibepp (см. `domain-graph.md`):

| Элемент | Описание |
|---------|----------|
| **Сущность `CoordinateMatch`** | Зафиксированная координата по оси протокола методики: `card_type`, `axis`, `coordinate`, `source_question_id`, `llm_interpretation_ref`, `created_at`. Append-only. Заменяет `CardSignal` из пороговой модели. |
| **Ребро** | `Answer` **maps_to** `CoordinateMatch` (детерминированный mapper по таблицам методики). |
| **Инвариант** | `Card` вычисляется только когда **протокол завершён** (для Cognitive v1 — все 12 ответов записаны как `protocol.coordinate_assigned`). Источник истины — правило (таблица соответствий методики); LLM-интерпретация — пользовательский опыт и аудит. |

### Сводка по фичам MVP (из vibepp `features`)

**В ядре MVP:** протокольный диалог по слою Core (Cognitive v1, 12 вопросов); неизменяемый event на ответ; LLM на каждом ответе показывает интерпретацию выбранной координаты, следующий вопрос берётся из очереди методики (LLM не генерирует); aeon_engine собирает карту по таблице соответствий после 12/12; safety до LLM; идемпотентность MAX; Book of Consciousness пополняется каждой завершённой сессией.

**Secondary (не блокируют ядро, но в плане продукта):** Glyph (SVG), кнопки в MAX, напоминание через 24 ч, `/profile`, `/history`.

**Вне MVP:** Mini App, NFT, Compatibility с другими пользователями, PDF, API для коучей/HR, языки кроме RU.

**Уточнение:** глиф — растровое изображение (DALL·E), не SVG; см. Блок 4–5.

## Блок 3. Диалог

**Статус блока:** обновлено по **ADR 002** — переход на протокольный диалог.

### Объём и ветвление

- **Длина сессии MVP:** **`MAX_QUESTIONS_PER_SESSION` = 12** — для Cognitive v1 совпадает с длиной протокола методики; для слоя Core длина **детерминирована протоколом**, не лимитом.
- **Ветвления сценария:** нет. Очередь вопросов фиксирована протоколом методики (Ц1 → Ц2 → … → Я3 для Cognitive v1). LLM **не генерирует** следующий вопрос на Core — он используется как интерпретатор предыдущего ответа.

### Форматы ответа пользователя (MVP)

См. vibepp `dialog.answer_formats` — `max_options` снято (ADR 002):

| Тип | Назначение |
|-----|------------|
| **Кнопки** | закрытые вопросы протокола; число вариантов берём из методики (8 для блока Ц Cognitive v1; 2–3 для других, где это допущено методикой); метки — «1»…«8» / «А»…«З» по нумерации в методике. |
| **Свободный текст** | если методика прямо допускает свободный ответ |
| **Шкала** | интенсивность, **диапазон 1–7** (для будущих карт; в Cognitive v1 не используется) |

Fallback при таймауте LLM-интерпретатора — короткое нейтральное подтверждение от шаблона + следующий протокольный вопрос (очередь детерминирована).

### «Не знаю», уход от темы, провокации

Обрабатывает **LLM-интерпретатор** — не как отдельные продуктовые ветки, а как уточнение в рамках текущего протокольного вопроса (просит конкретный пример, не двигает протокол вперёд до получения интерпретируемого ответа). Тон и границы — общие в системной части `cognitive-interpret-answer@v1`.

### Персона

Оставляем как в vibepp: имя **ÆON**, тон и границы (`dialog.persona`), safety-триггеры и действие — `dialog.safety`.

## Блок 4. LLM-ядро

**Статус блока:** зафиксировано.

### Провайдеры и роли

| Роль | Сервис |
|------|--------|
| Основной диалог, следующий вопрос | **Anthropic Claude (API)** |
| Резерв при недоступности/ошибке основного | **OpenAI** (fallback-модель; конкретное имя — в конфиге на этапе реализации) |
| Визуал глифа | **OpenAI Images (DALL·E)** — отдельно от текстового ядра |

Интеграции версионировать (API-ключи, модели) в конфиге; в event-логе фиксировать провайдера и имя модели.

### Архитектура вызовов (протокольный режим, ADR 002)

Шаги на ход пользователя (после фиксации `answer.given`):

1. **Protocol Mapper (детерминированный)** — выбор пользователя по протокольному вопросу → координата по таблицам методики (для Cognitive v1 — §§3.2–3.4). Без LLM. Выход: одно событие **`protocol.coordinate_assigned`** на ответ.
2. **LLM-интерпретатор (Claude, prompt `cognitive-interpret-answer@v1`)** — формулирует короткое объяснение пользователю, к какой координате его относит mapper и почему. Записывается **`llm.called` (purpose=answer_interpretation)** + **`answer.interpreted`**.
3. **Dialog Engine** добавляет следующий вопрос из очереди протокола в конец сообщения LLM-интерпретатора → одно сообщение пользователю «интерпретация + следующий вопрос» → **`question.asked`**.
4. **На завершении протокола (12/12 ответов)** — `card.computed` по правилу-mapper (LLM-сверка снята; см. ADR 002).

**Цель:** одно тяжёлое LLM-обращение на ход (интерпретатор) и одно — на финале. Mapper детерминированный и быстрый; общая цепочка укладывается в `MAX_BOT_REPLY_TIMEOUT_SEC`.

Все вызовы — `llm.called` с полем `purpose`; fallback OpenAI логируется аналогично.

### Память между сессиями (что куда)

| Слой | Содержание | Назначение |
|------|------------|------------|
| **Сжатый профиль** | Карты, confidence, рисунок типов, архетип. | Контекст «**кто ты сейчас**» для промпта |
| **Координаты текущей сессии** | `CoordinateMatch` по уже отвеченным вопросам протокола | Состояние диалога; восстановление при возврате |
| **Хвост ответов** | **Последние 3–5** ответов и их интерпретаций | Тональная непрерывность LLM-интерпретатора |

**Book of Consciousness:** храним и показываем пользователю, используем в метриках; **в системный промпт LLM не подмешиваем** — в промпт идут сжатый профиль + хвост ответов + текущая позиция в протоколе.

### Safety (два рубежа)

Одного «классификатора перед дорогим вызовом» по текущему vibepp **недостаточно** — нужно явное разделение:

- **Gate 1 — до всего дорогого:** отдельный модуль в **`src/stability/`** (свои тесты): **rule-based** и/или **очень лёгкий классификатор** (ключевые слова, паттерны). Стоимость ~0 по сравнению с LLM. При срабатывании — **стоп**, ответ по сценарию кризиса, **без вызова** Claude/OpenAI fallback.
- **Gate 2 — в основном LLM:** границы и запреты в **системном промпте** основного диалога; второй рубеж для edge cases.

Концептуальные триггеры и действия остаются согласованы с `dialog.safety` в vibepp; архитектурно фиксируем **Gate 1 + Gate 2** как обязательную схему.

## Блок 5. Результат на выходе

**Статус блока:** зафиксировано.

### Формат MVP (достаточно для продукта)

- **Текст в чате** + **картинка** (глиф). Отдельный веб-экран / HTML **не нужны**.

### Конец сессии: сразу, двумя сообщениями

Весь ценностный вывод — **в конце сессии**, не «часть сейчас — часть завтра». Между двумя сообщениями **пауза 2–3 секунды** (задержка на бэке перед вторым отправлением).

| Шаг | Когда | Содержание |
|-----|--------|------------|
| **Сообщение 1** | Сразу | **Глиф** + **одна фраза-суть** (эмоциональная точка). Пример формата: «Ты — Строитель-Аналитик. Твоя сила в том, что ты делаешь с пониманием системы.» |
| **Сообщение 2** | Через **~3 с** | **Краткий профиль:** **3–4 текстовые карточки** — архетип, когнитивный стиль, зона роста, энергия. Только текст, **без HTML**. |

### Через 24 часа (после завершённой сессии)

Одно **push-уведомление в MAX**, не общее «вернись», а **персонализированный крючок из данных сессии**, например:

«Вчера ты прошёл Core Layer. Хочешь углубиться в [конкретная карта с наибольшим напряжением]?»

Карта и формулировка берутся из **фактов сессии** (сигналы/напряжение по картам).

### v1 (не MVP)

- **Еженедельный глиф-дайджест:** что изменилось в профиле, какие сигналы накапливаются между сессиями.

### Шеринг (MVP)

- **Минимум один обязательный формат:** **PNG-карточка**, генерируется на **бэке**: глиф + имя архетипа + одна строка сути.
- Пользователь пересылает как **обычное медиа** в MAX или в другой мессенджер — **без обязательных ссылок**.
- **Техника:** шаблон **SVG** на бэке → растр **PNG** через **sharp** + **@resvg/resvg-js** (без headless-браузера).
- **Ссылка на веб-профиль** — **v1**, когда появятся Mini App или лендинг; в MVP ссылок на внешние страницы **нет** (низкая гарантия перехода; картинка работает везде).

### Триггеры и команды (как в vibepp)

Оставляем: напоминание о **незавершённой** сессии через 24 ч, команды **`/profile`**, **`/history`**. Дополнительно к ним — описанный выше **push через 24 ч после завершённой** сессии (отдельный продуктовый триггер).

## Блок 6. MAX-платформа

**Статус блока:** зафиксировано.

### Аккаунт бота

Бот в MAX **уже создан**.

### Webhook (MVP, без альтернатив)

- **Только webhook** в проде. **Long polling** — допустим как **костыль для локальной разработки**, не целевой режим для production.
- Webhook предпочтителен (как в Telegram-подобных API): ниже задержка, нет холостых запросов, нормальное масштабирование.

### Требования к endpoint

- **HTTPS обязателен**; **самоподписанный** сертификат **не подойдёт** — нужен **валидный TLS**.
- **Фиксированный IP не требуется**; нужен **стабильный домен** для URL webhook.
- **Порт:** типично **443**, **80**, **88** или **8443** — **уточнить в актуальной документации MAX Bot API** для выбранной версии API.

### Поверхность продукта в MVP

Только **чат** (Mini App / веб — **v1**, см. Блок 5).

### Заметка по стеку (связь с MAX)

Примеры и сценарии HTTP к MAX удобно опираются на **TypeScript / Node** (в т.ч. материалы вокруг dev.max.ru / экосистема); детали стека — в разделе **«Решения по стеку»** ниже.

## Блок 7. Инфраструктура и бизнес

**Статус блока:** зафиксировано (нагрузка, приватность, процесс; монетизация — **Блок 9**).

### Деплой и dev (как ранее)

- **Домен + SSL:** Let's Encrypt (**Caddy** или **nginx**).
- **Сервер** с публичным IP; **РФ-облака:** Yandex Cloud, VK Cloud, Selectel.
- **Caddy** как reverse proxy.
- **Локально:** **ngrok** / **cloudflared**; в **docker-compose** — опциональный сервис туннеля.

### Нагрузка и масштабирование (MVP)

- **Честный старт:** порядка **сотен MAU**; **тысячи** — если есть **канал с аудиторией** на запуск.
- **Архитектурное решение:** один процесс **Node.js** + **PostgreSQL** на одном сервере **выдерживает порядка до ~5k MAU** без горизонтального масштабирования (оценка порядка величины, не SLA).
- **Kubernetes в MVP не закладываем** — преждевременная оптимизация для описанного масштаба.

### Приватность и данные

- Уровень: **средний / серьёзный продукт**, **GDPR-like**:
  - право на удаление — **автоматизировано** (процесс + технический контур);
  - **PII отделены** от аналитики;
  - **данные только в РФ** (хостинг и хранилища в юрисдикции РФ).

### Команда и процесс разработки

- **Соло** + агентная разработка в **Cursor**; репозиторий **открытый** на GitHub.
- **GitHub Issues, Projects, Milestones, Labels** — основной трекинг без отдельной «команды».
- Практика: roadmap → итерации → **issues** с `closes #N` в PR; история в комментариях к issue; будущий контрибьютор онбордится по репо и Issues.

**Артефакт:** полный набор черновиков GitHub Issues по итерациям — `docs/GITHUB_ISSUES.md`.

### Git: коммиты и ветки

Полная стратегия (когда коммитить, частота, `feat/iter-N`, `main` = рабочее и задеплоено) — **`docs/GIT_STRATEGY.md`**.

## Блок 9. Монетизация, QIP и биллинг

**Статус блока:** решения по модели зафиксированы; детали интеграции с QIP — уточняются при подключении кошелька/API.

### Вопросы, на которые отвечает этот блок

1. Какая модель дохода в MVP и почему?
2. Как не сломать Event Core и PDA при платных действиях?
3. Что отложено после MVP?

### Зафиксированные решения

**Модель MVP: freemium через QIP Coin**

- **Бесплатно:** базовый профиль (границы «базового» описать в `constants.md` при реализации — какие слои/глубина входят без оплаты).
- **За QIP Coin (покупка монет):** **глубокие слои** и **расширенный анализ** — органично к существующей механике наград QIP.

**Альтернативы (не MVP):**

| Модель | Плюс | Минус / когда не MVP |
|--------|------|------------------------|
| Pay-per-profile | низкий барьер | слабее retention |
| Подписка | LTV | нужна регулярная ценность (дайджест, динамика) — в MVP **ещё нет** |

### Архитектурный принцип

- Монетизация — **первоклассный контур**: платные действия отражаются **событиями** (например списание QIP, разблокировка платного слоя — конкретные `event_type` зафиксировать в `events.md` при реализации).
- **Stability Engine** знает, разрешён ли пользователю слой/фича (баланс / покупка / freemium-лимит).
- **Инвариант:** списание или разблокировка не обходят event log; нет «тихого» доступа к платному без цепочки событий.

### Открыто до реализации

- Точные **цены в QIP** за слой / сессию / отчёт.
- **API кошелька / платёжный поток** QIP (webhook оплаты, баланс пользователя).
- Юридическое оформление оферты и учёт цифровых товаров — вне скоупа SPEC, но блокирует прод-приём платежей.

## Решения по стеку (Этап 1)

**Статус:** зафиксировано (Блок 6 + согласование стека).

| Компонент | Выбор |
|-----------|--------|
| Язык | **TypeScript 5.x** |
| Runtime | **Node.js 22 LTS** |
| Веб-фреймворк | **Fastify** |
| Тесты | **Vitest** + **fast-check** (property-based для инвариантов; Hypothesis-уровня в JS нет — риск приемлемый) |
| Миграции БД | **node-postgres** + **db-migrate** *или* **Drizzle ORM** (уточнение при старте репо) |
| PNG из SVG (шеринг-карточка) | **sharp** + **@resvg/resvg-js** (без Playwright/cairosvg) |

**Почему TypeScript для этого проекта (кратко):** один язык на webhook, dialog engine, LLM-клиенты и вспомогательные сервисы; официальный **Anthropic SDK** для TS; event-driven + async/await нативно в Node; примеры клиента к Bot API без сюрпризов в TS-экосистеме; быстрее вывести MVP, чем типичный бойлерплейт Java/Go для той же задачи.

**Без изменений по смыслу:** PostgreSQL, Redis, event store в Postgres, Docker Compose для локальной разработки, OpenTelemetry / метрики / JSON-логи — как в исходном промпте разработки.
