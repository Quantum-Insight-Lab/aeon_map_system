# ADR 002: Протокольный режим карт (Cognitive v1) и отказ от пороговой модели сигналов

## Контекст

В исходной SPEC для слоя Core (iter-4) был зафиксирован **порогово-непрерывный** контур:

- сквозной **CardSignal** на каждый `answer.given` (cross-layer routing),
- карта (`card.computed`) выпускается, как только сошлись пороги
  `MIN_AGGREGATE_SIGNAL_WEIGHT_FOR_CARD` и `CARD_CONFIDENCE_THRESHOLD`,
- диалог адаптивный, без фиксированного списка вопросов.

С появлением методики **ÆON Cognitive Identity Map v1**
(`docs/methodology/Cognitive_Identity_Map_v1.md`) для слоя Core доступен
**протокольный** способ закрытия неопределённости: 12 фиксированных
вопросов (4 × Ц + 5 × М + 3 × Я), детерминированный маппер «выбор → координата»
(§§3.2–3.4), сборка типа из таблицы соответствий §4.1 и алгоритма §4.2.

Пороговый и протокольный контуры — две разные дисциплины снижения
неопределённости. Совмещать их в продукте — путь к разным семантикам
`confidence` и неконсистентному пользовательскому опыту.

---

## Решение

### 1. Протокольный режим — единственный для карт

Все карты ÆON Map переходят на **протокольный режим** по мере появления
своих методик. Пороговая модель сигналов (CardSignal / агрегат веса /
условный выпуск карты по достижению порогов) — **исключена из продукта целиком**,
а не «помечена устаревшей для Cognitive».

- В iter-4 в Core MVP остаётся **только** `CognitiveIdentityMap` (по протоколу v1).
- `BehavioralPatternMap` переносится в iter-5 / v1 — до появления его
  собственной методики и протокола.
- Сквозной **signal routing** между слоями (SPEC Блок 2, §«Сквозные связи»)
  **временно отключён**. Возможность вернуть его рассмотрим после стабилизации
  протоколов нескольких карт; до этого момента ответ протокола **не**
  пополняет «копилки» других карт.

### 2. Источник истины при назначении карты

При расчёте карты Cognitive основным судьёй выступает **детерминированное правило**
(таблица §4.1 + алгоритм §4.2 методики). LLM используется на каждом ответе как
**интерпретатор**: показывает пользователю, к какой координате его относит,
и почему. Финальное расхождение между правилом и LLM-интерпретацией
**не блокирует** выпуск карты — оно фиксируется как наблюдение в
**Book of Consciousness** (через соответствующее событие записи сессии).

### 3. Константы

- **Удалены:** `MIN_ANSWERS_PER_LAYER`, `MIN_AGGREGATE_SIGNAL_WEIGHT_FOR_CARD`.
- **Сохранены:**
  - `MAX_QUESTIONS_PER_SESSION = 12` (для Core MVP совпадает с длиной протокола);
  - `CARD_CONFIDENCE_THRESHOLD = 0.72` — переопределяется как **мера согласованности**:
    степень совпадения координат с ближайшим типом из таблицы §4.1
    и/или уровень согласия LLM-интерпретации с правилом. Численная формула —
    в `docs/SPEC/constants.md` после реализации iter-4.
- **Добавлены:**
  - `COGNITIVE_PROTOCOL_VERSION = v1` — фиксирует версию методики, по которой
    собрана конкретная карта (для аудита и миграций при v2);
  - `LLM_RULE_AGREEMENT_THRESHOLD` — нижняя граница согласия LLM-интерпретации
    с правилом, ниже которой расхождение записывается в Book of Consciousness
    (значение калибруется после первых сессий).

### 4. Инварианты

- **INV-03** — сохранён: `Card` назначается только при `confidence ≥ CARD_CONFIDENCE_THRESHOLD`
  (с новой семантикой `confidence`, см. §3 выше).
- **INV-04** — сохранён: `Card` не в двух взаимоисключающих состояниях без явного
  «синтетического» флага. Множественные типы из методики §4.3 — это **рисунок**
  и должны помечаться явно в payload `Card`.
- **INV-05** — переформулирован: **никакая** карта не вычисляется до завершения
  её протокола. Для Cognitive в iter-4 это 12 / 12 ответов протокола v1.
  Старое условие про `MIN_ANSWERS_PER_LAYER` снято вместе с константой.
- **INV-09** — удалён: пороговая модель веса `CardSignal` исчезает.
- **INV-10 (новый)** — для каждого ответа Core событие `answer.interpreted`
  записывается **до** следующего `question.asked` (аналог INV-06 для шага
  LLM-интерпретации; гарантирует, что пользователь не получит следующий вопрос,
  пока интерпретация предыдущего ответа не зафиксирована в event store).

### 5. События

- **`card_signal.received`** — удалён.
- **`protocol.coordinate_assigned` (новый)** — фиксирует результат
  детерминированного маппера на каждом протокольном ответе:
  `axis ∈ {goal, modality, anchor}`, `coordinate`, `source_question_id`,
  `llm_interpretation`, `llm_call_id`. Causation — `answer.given`.
- **`answer.interpreted` (новый)** — фиксирует, что LLM сформулировал
  и отправил пользователю интерпретацию его ответа (текст показанного
  объяснения). Записывается до отправки сообщения пользователю и до
  следующего `question.asked` (INV-10).
- **`card.computed`** — сохранён; payload расширяется полями
  `protocol_version`, `coordinates {goal, modality, anchor}`,
  `matched_types[]`, `disagreement_with_llm` (опционально).
- **Отдельный `protocol.completed` не вводится** — сигнал о завершении
  протокола несёт сам факт успешной записи `card.computed`.

### 6. Доменная модель

- Сущность `CardSignal` **переименована** в `CoordinateMatch` и переосмыслена:
  это не «вес сигнала», а зафиксированная координата по оси протокола
  (`axis`, `coordinate`, `source_question_id`, `llm_interpretation_ref`,
  `created_at`). Append-only.
- Ребро `Answer contributes_signal_to Card` заменяется на
  `Answer maps_to CoordinateMatch` (детерминированно через маппер).

### 7. Архитектура

- **`signal_classifier` → детерминированный mapper:** «выбор пользователя
  по протокольному вопросу → координата по таблицам §3.2–3.4 методики».
  Никакой LLM-классификации на этом шаге.
- **`dialog_engine`:** для протокольного слоя следующий вопрос **берётся
  из очереди протокола** (Ц1 → … → Я3), но **доставляется пользователю
  в конце сообщения LLM-интерпретатора** — единым ответом «интерпретация
  предыдущего ответа + следующий вопрос».
- **`llm_client`:** добавляется новая роль «интерпретатор протокольного
  ответа» с отдельным версионированным prompt (§9 ниже).
- **`aeon_engine`:** на завершении протокола читает `CoordinateMatch`-и сессии,
  применяет таблицу §4.1 и алгоритм §4.2, сверяет с LLM-интерпретациями,
  выпускает `card.computed` (и фиксирует расхождение, если оно превышает
  `LLM_RULE_AGREEMENT_THRESHOLD`).
- **`data_flow`** в vibepp переписывается под протокольный сценарий
  (без CardSignal-копилки и без LLM-генерации следующего вопроса на Core).

### 8. Слои и форматы ответов

- `aeon_layers.mvp.I` — `cards: [CognitiveIdentityMap]`,
  `min_questions: 12` (по протоколу v1); `BehavioralPatternMap` — вне MVP-Core.
- `dialog.answer_formats`:
  - ограничение `max_options` **снято**;
  - для вопросов с 8 вариантами (блок Ц) пользователю отдаются кнопки
    с подписями вида **«1»**, **«2»**, …, **«8»** или **«А»**, **«Б»**,
    **«В»** и т. д. — по нумерации/буквенному коду варианта в методике;
  - для вопросов с 2 вариантами (где методика прямо это допускает —
    см. соответствующие места в `Cognitive_Identity_Map_v1.md`) —
    кнопки на 2 варианта без принудительного набора до 4.
  - формат `scale 1–7` сохраняется для будущих карт; в Core v1 не используется.

### 9. Промпты

- **`prompts/dialog-next-question@v1` — выводится из эксплуатации** для Core.
  LLM-генерация следующего вопроса из iter-3 (`core:llm:k`)
  оборачивается **feature-flag** (по умолчанию выключен на Core),
  чтобы код не пришлось удалять немедленно и можно было использовать его
  для отладочных сравнений. На Core следующий вопрос даёт протокол.
- **`prompts/cognitive-interpret-answer@v1` — новый, обязательный для iter-4.**
  Назначение: LLM получает выбор пользователя по протокольному вопросу,
  координату от детерминированного маппера и формулирует короткую
  интерпретацию для пользователя в духе методики (§3 «Подсказки в вопросах
  видны намеренно»). К этому ответу Dialog Engine добавляет следующий
  вопрос из очереди протокола.

### 10. Iter-4 и iter-3

- `iterations[id=4]` переписывается под протокольный путь:
  deliverables — протокол v1, mapper, prompt-интерпретатор, новые события,
  INV-03/04/05/10 с property-based тестами, `card.computed` после 12 / 12.
- В iter-3 LLM-next-question остаётся в кодовой базе под feature-flag;
  отдельной чистки в этом ADR не предписывается.

### 11. Инвариант источника истины

В случае разногласия формулировок между документами приоритет уточнения
сохраняется по `AGENTS.md` §1: **vibepp.yaml** — машинный канон,
затем `.md`-артефакты. Этот ADR — основание для синхронизации vibepp
и SPEC; до синхронизации — этот ADR главнее текста SPEC и Issue 5.

## Последствия

- **Документация:** требуется синхронизация
  `vibepp.yaml`, `docs/SPEC/*`, `docs/SPEC_SUMMARY.md`, `docs/GITHUB_ISSUES.md`
  (Issue 5) под пункты §§ 1–10. Делается отдельным изменением, не в коммите ADR.
- **Прежние решения:** PR-ы iter-1 / iter-2 / iter-3 в репозитории остаются
  в силе; LLM-генерация следующего вопроса (iter-3) уходит под feature-flag,
  а не удаляется.
- **Тесты:** старые тесты на INV-09 и `MIN_AGGREGATE_SIGNAL_WEIGHT_FOR_CARD`
  снимаются вместе с инвариантом и константой; новые property-based тесты
  ставятся на INV-05 (новая формулировка) и INV-10.
- **Возврат сквозного routing:** возможен после стабилизации протоколов
  ≥ 2 карт; решение оформляется отдельным ADR.
